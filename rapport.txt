\begin{abstract}
This paper presents the design, implementation, and evaluation of a full-stack blockchain system built from scratch in Go with a React-based frontend explorer. The system implements SHA-256-based block linking, configurable Proof-of-Work consensus, ECDSA-signed transactions with nonce-based replay protection, a transaction mempool, and a decentralized peer-to-peer networking layer using UDP broadcast discovery and TCP communication. We evaluate the system through four experiments: mining performance under varying PoW difficulty, chain integrity under tampering, transaction validation under adversarial conditions, and multi-node synchronization behavior. Results show exponential mining-time growth with difficulty, successful detection of all chain tampering attempts, rejection of five distinct attack vectors (replay, signature forgery, public-key spoofing, overdraft, and future-timestamp manipulation), and correct chain convergence across networked nodes. The system demonstrates that a modular, relatively compact implementation can faithfully reproduce the core security and consensus properties of production blockchains while remaining accessible for experimentation and education.
\end{abstract}

\section{Introduction}

Blockchain technology has emerged as a fundamental component of decentralized systems, 
enabling trustless data exchange without reliance on centralized authorities. However, 
while blockchain systems are widely studied, their core mechanisms are often presented 
through simplified theoretical models that abstract away critical implementation details. 
As a result, there is a limited understanding of how these systems behave in practice, 
particularly with respect to networking, synchronization, and system-level trade-offs.

In particular, the interaction between consensus mechanisms, transaction propagation, 
and peer-to-peer communication remains difficult to study without a concrete implementation. 
Many educational models omit networking entirely, while production systems introduce 
significant complexity that obscures fundamental design principles. This creates a gap 
between conceptual understanding and practical system behavior.

In this work, we bridge this gap by designing and implementing a full-stack blockchain 
application that captures the essential components of a decentralized system while 
remaining accessible for experimentation. The system consists of a peer-to-peer blockchain 
node implemented in Go, combined with a web-based frontend built in React for interaction 
and visualization.

A central aspect of our design is the networking layer. Nodes operate in a fully 
decentralized environment, discovering peers via UDP-based broadcast and establishing 
TCP connections for reliable communication. This approach enables nodes to propagate 
transactions, synchronize newly mined blocks, and maintain a shared view of the blockchain 
without centralized coordination. By explicitly separating lightweight discovery from 
reliable data transfer, the system exposes key trade-offs between efficiency and reliability 
in distributed systems.

To support interaction and observability, we integrate a frontend interface that allows 
users to visualize the blockchain, monitor node states, initiate mining, and submit 
transactions. This enables an end-to-end workflow, from user-generated transactions to 
block creation and network-wide propagation.

At a high level, we make the following contributions:
\begin{itemize}
  \item Design and implementation of a modular blockchain system in Go, including 
  block management, transaction handling, Proof-of-Work, and validation mechanisms.
  \item A decentralized peer-to-peer networking layer based on UDP discovery and TCP 
  communication, enabling dynamic node discovery and synchronization.
  \item A web-based frontend for interacting with the system and visualizing blockchain state.
  \item An experimental evaluation of system behavior, including, chain integrity, transaction validation under adversarial conditions, and multi-node synchronization.
\end{itemize}

This paper addresses the following research questions:
\begin{itemize}
  \item What architectural trade-offs are required to balance technical realism with educational clarity when implementing an interactive blockchain prototype?
  
  \item To what extent do cryptographic mechanisms, including ECDSA signatures, nonce ordering, and timestamp validation, protect against transaction, level attacks such as replay and forgery?
  
  \item What limitations arise from the use of a simplified peer-to-peer networking model?
\end{itemize}

\section{Background}
Traditional digital systems rely heavily on a trusted central authority, such as a bank or a single server, which introduces points of failure, censorship risks, and concentrated control. To eliminate this dependence, a blockchain serves as a distributed ledger that maintains a continuously growing list of records, called blocks, without relying on any central power source. Introduced by Nakamoto \cite{nakamoto2008bitcoin} as the underlying architecture for a peer-to-peer electronic cash system, this decentralized design enables a completely trustless environment. Instead of relying on a central intermediary, each participant in the network maintains a local copy of the ledger and verifies updates independently. Correctness and data integrity are ultimately ensured through a combination of cryptographic mechanisms and consensus protocols, allowing multiple nodes to agree on a shared state while tolerating faulty or malicious participants.

\subsection{Blockchain Structure}

The core of a blockchain is the structure of its blocks and the way they are linked together \cite{narayanan2016bitcoin}. Each block contains a set of transactions, a timestamp, and a cryptographic hash of the previous block. By including the previous hash, blocks are chained together in a sequential manner, forming a tamper-evident data structure. Any modification to a block would change its hash, thereby invalidating all subsequent blocks in the chain. This property ensures immutability, as altering historical data would require recomputing the hashes of all following blocks, which becomes computationally infeasible in practice \cite{nakamoto2008bitcoin}.

\subsection{Cryptographic Hashing}
Cryptographic hash functions map arbitrary-length input to a fixed-length output with three key properties: determinism (the same input always produces the same hash), avalanche effect (a small change in input produces a drastically different output), and pre-image resistance (it is computationally infeasible to reverse the hash). SHA-256, a member of the SHA-2 family standardized by NIST, produces a 256-bit (32-byte) digest \cite{nist_sha256}. It is the primary algorithm used in Bitcoin and remains collision-resistant under current computational capabilities \cite{nakamoto2008bitcoin}.

\subsection{Proof-of-Work}
The concept of using a computational hurdle as an artificial cost to deter resource abuse was originally proposed by Dwork and Naor \cite{dwork1992pricing}. In blockchain systems, this principle is adapted into Proof-of-Work (PoW), a consensus mechanism that requires miners to expend computational effort to produce a valid block \cite{nakamoto2008bitcoin}. Nodes connected to the network compete simultaneously to solve a cryptographic puzzle, which involves finding a block hash that satisfies a predetermined difficulty target. The first miner to discover a valid hash wins the right to append the next block to the blockchain and broadcast it to the network.

This process demands immense computational power, as miners must execute a brute-force search by iteratively changing a variable called a nonce until the resulting hash meets the strict structural criteria. Because cryptographic hashing functions are entirely deterministic yet unpredictable, there are no mathematical shortcuts. Therefore, the probability of finding a solution is directly proportional to the amount of computational work a node can perform. Consequently, this computational hurdle serves as a rate-limiting defense mechanism, making the modification of historical data economically and computationally prohibitive for malicious actors.

\subsection{Transactions}
In a blockchain architecture, a transaction represents a cryptographically secured transfer of value between two digital accounts. Rather than relying on physical identities, users interact with the network through cryptographic wallets, which consist of a public-private key pair. The wallet address is derived directly from the user's public key, serving as their public identity on the network, while the private key is kept confidential and used to authorize actions.

To ensure that funds can only be moved by their rightful owner, transfer transactions are digitally signed using public-key cryptography. When a user initiates a transfer, their wallet uses the private key to generate a unique digital signature over the transaction details, which include the recipient's address, the amount, and a sequence number or nonce. 

However, cryptographic signatures alone cannot prevent a sender from signing multiple conflicting transactions using the same funds. To thwart such double-spending attacks without a trusted central intermediary, all transactions must be publicly announced, requiring a mechanism for network participants to agree on a single, chronological history. This is achieved by organizing verified transactions into a linear, shared chain of blocks \cite{nakamoto2008bitcoin}. Consequently, a transaction is universally validated only when independent nodes collectively verify its chronological position against their local copy of the ledger, establishing a decentralized consensus that rejects any conflicting historical records. 


\subsection{Mempool}
Once transactions are broadcast to the network, they require temporary storage before being mined \cite{binance_mempool}. The mempool (memory pool) acts as this staging area for unconfirmed transactions. In blockchain networks, when a node receives a new transaction, it validates the transaction against network consensus rules before adding it to its local pool. Miners then select transactions from this pool to bundle into new blocks. This decoupling between transaction submission and block creation is a fundamental architectural pattern in production blockchain systems.

\subsection{Peer-to-Peer Networking}
In a blockchain network, nodes communicate peer-to-peer (P2P) without relying on a central coordinator. To maintain the distributed ledger, nodes autonomously discover active peers and establish communication channels to propagate unconfirmed transactions and newly mined blocks. However, propagation is trustless, nodes independently validate all received transactions and blocks against network consensus rules before accepting or relaying them.

Due to network latency, nodes may receive competing blocks simultaneously, leading to temporary chain branches. To resolve this, networks employ a fork-resolution mechanism, typically the "longest chain rule," where nodes always consider the valid chain with the greatest accumulated proof-of-work to be the correct one. Additionally, the gossip-based protocol is inherently fault-tolerant. Broadcasts do not need to reach all nodes instantly, and nodes that miss broadcasts can request missing blocks from peers once they reconnect or detect a gap in their chain height \cite{nakamoto2008bitcoin}.

\subsection{Consensus}
In a decentralized blockchain network, consensus protocols ensure that all independent nodes synchronize and agree on a single, authoritative version of the ledger without relying on a central coordinator. A foundational mechanism for achieving this in Proof-of-Work networks is the longest-chain rule \cite{nakamoto2008bitcoin}.

Because multiple miners work on solving cryptographic puzzles simultaneously, the network will occasionally experience a temporary split, known as a fork, where two different valid blocks are discovered at roughly the same time. To resolve these conflicts and maintain a unified history, nodes inherently trust the chain that represents the greatest amount of accumulated proof or computational work. When a node detects that a peer possesses a longer valid chain, it will fetch the missing blocks and perform rigorous validation. This validation includes checking the cryptographic hashes, block sequencing, and individual transaction states. If the new chain is fully verified, the node adopts it, discarding its own shorter branch. This automatic realignment ensures that temporary network partitions are short-lived, forcing all honest nodes to quickly converge back onto a single, authoritative timeline. 


\subsection{Token Distribution Mechanisms}
In blockchain networks, initial token distribution determines how cryptocurrency is introduced into circulation. While production networks often rely on continuous, algorithmic emission via mining rewards, alternative mechanisms are frequently utilized for bootstrapping networks or in test environments.

\subsubsection{Pre-mining}
Pre-mining refers to the process where a specific amount of a cryptocurrency's total supply is minted or allocated before the public launch of the network \cite{binance_premining}. This approach is commonly used to provide immediate ecosystem liquidity or fund initial core project entities.

\subsubsection{Crypto Faucets}
A crypto faucet is an application or mechanism that distributes small amounts of cryptocurrency to users, typically without requiring financial compensation \cite{binance_faucet}. In development networks and testnets, faucets serve as critical infrastructure, enabling developers and testers to acquire tokens required for system interaction without economic cost.

\subsection{Core Ledger and Consensus}
At the foundation of the system is the blockchain core, which manages the ordered sequence of blocks and maintains the global state, including account balances, nonces, and processed transaction IDs. It enforces strict chain validation rules, executing genesis verification and stateful transaction checks across the entire ledger. Pending transactions are staged in a thread-safe mempool utilizing mutex-protected data structures to handle concurrent operations safely. Block finality is achieved through a Proof-of-Work (PoW) consensus mechanism, where miners iteratively compute SHA-256 hashes to satisfy a deterministic difficulty prefix.

\subsection{Cryptography and Transaction Model}
The identity and authorization layer is governed by Elliptic Curve Cryptography (ECDSA) utilizing the P-256 curve. Wallet addresses are derived via SHA-256 hashing of the uncompressed public key, truncated to a 20-byte identifier. The transaction system handles the creation of genesis, coinbase, and standard transfer transactions. To ensure data integrity and non-repudiation, transfer transactions are secured with fixed-width 64-byte $(r,s)$ signatures, validated against the canonical byte sequence of the transaction payload.

\subsection{Peer-to-Peer Networking}
Decentralized communication is handled by a custom P2P networking implementation. The architecture employs a dual-protocol approach: lightweight UDP broadcasts (emitted every three seconds) are used for local peer discovery, while reliable TCP connections manage continuous data exchange. Upon a successful TCP handshake, peers exchange known addresses to facilitate transitive discovery. The network state is synchronized using an inventory-based \texttt{inv/getdata} protocol for blocks and transactions, with strict limits on connection counts (e.g., a maximum of six peers) to prevent resource exhaustion.

\subsection{Application Interface and Analysis}
To bridge the internal blockchain logic with external users, the system exposes a comprehensive HTTP REST API. This interface provides endpoints for state queries, mempool access, transaction submission, and mining triggers. Interfacing with this API is \textit{BlockView}, a React-based frontend explorer that enables real-time chain visualization and node management. Finally, the system design incorporates an isolated security testing module. This suite programmatically executes adversarial scenarios including replay attacks, signature forgery, and state overdrafts against a fresh blockchain instance to actively verify the network's cryptographic resilience.

\subsection{Initial Token Distribution and Faucet Implementation}
To facilitate immediate transaction validation and system testing within our development environment, the architecture implements the distribution mechanisms described in the background.

Upon initialization, the blockchain core generates a hardcoded Genesis block. This block contains a specialized coinbase transaction that executes the pre-mining phase by minting a predetermined supply of tokens. These tokens are allocated directly to a centralized treasury address, as well as to several pre-configured test accounts (e.g., "Alice" and "Bob"). This setup allows users to immediately interact with the system, execute peer-to-peer transfers, and observe validation logic upon startup without the need to initialize multiple nodes or perform initial mining.

To distribute tokens from the treasury to new external accounts, the system exposes a Faucet via the \texttt{POST /faucet} HTTP API endpoint. When an external account requests funds, the Faucet component constructs and signs a standard transfer transaction from the treasury to the user's derived wallet address. This transaction is then broadcast to the mempool, ensuring that the distribution of funds is subject to the exact same cryptographic validation and mining sequence as standard peer-to-peer transactions.


\section{Implementation}
This section details the practical realization of our decentralized blockchain system, outlining the structures, protocols, and architectural decisions made during development. The core blockchain node, state machine, cryptographic validation, and peer-to-peer networking modules are implemented entirely in Go, leveraging its native support for concurrency and robust standard networking library. The interactive user interface, real-time block explorer, and isolated security evaluation panel are realized as a modern single-page frontend application utilizing React.js, compiled and served via the Vite build tool. Together, these layers form a cohesive, runnable system that demonstrates the security and decentralized synchronization properties of a production-like blockchain.


\subsection{Block Structure and Hashing}

Each block is represented as a Go struct containing seven fields: 

\begin{description}
    \item[\texttt{Index}:] Sequential position in the chain.
    \item[\texttt{Timestamp}:] Unix time in milliseconds.
    \item[\texttt{Transactions}:] A slice of \texttt{Transaction} structs.
    \item[\texttt{PrevHash}:] The SHA-256 hash of the preceding block.
    \item[\texttt{Hash}:] The block's own computed hash.
    \item[\texttt{Nonce}:] The Proof-of-Work solution.
    \item[\texttt{Difficulty}:] The required number of leading zero hex characters.
\end{description}

The hash is computed by serializing the relevant block fields into a deterministic pipe-delimited string:
\begin{verbatim}
record = fmt.Sprintf("%d|%d|%s|%s|%d|%d",
    block.Index, block.Timestamp,
    json.Marshal(block.Transactions),
    block.PrevHash, block.Nonce,
    block.Difficulty)
hash = SHA-256(record)
\end{verbatim}

This approach ensures that any modification to any field, including the transaction contents, produces a completely different hash, providing immediate tamper evidence. By serializing the entire transaction array directly into the block record, our implementation deliberately simplifies the architecture. It bypasses the need for Merkle Trees, commonly used in production networks like Bitcoin in favor of a more straightforward validation model suitable for our scope.


\subsection{Proof-of-Work Implementation}
The Go-based blockchain integrates a Proof of Work (PoW) consensus mechanism where nodes computationally compete to find a SHA-256 block hash meeting a strict difficulty prefix. This mechanism fundamentally protects the network from data tampering and ensures stable node synchronization. To satisfy the consensus rules, a mutable nonce ($N$) is iteratively incremented with one and hashed alongside the block's core attributes: Index ($I$), Millisecond Timestamp ($TS$), JSON-serialized Transactions ($TX$), Previous Block Hash ($P$), and Difficulty ($D$), concatenated deterministically using a pipe delimiter ($|$). The mining loop continues until the resulting hexadecimal hash exhibits a prefix of at least $D$ consecutive leading zeros, which mathematically constrains the hash value below a target threshold determined by the difficulty:
\begin{equation*}
    H(B) = \text{SHA-256}(I \mid TS \mid TX \mid P \mid N \mid D) < 16^{64 - D}
\end{equation*}

\subsection{Transaction Signing Workflow}
When a user initiates a transaction via a wallet, the system executes the following steps in \texttt{NewSignedTransferTransaction}:
\begin{enumerate}
  \item \textbf{Canonical Serialization:} The transaction attributes, consisting of the sender's address (\texttt{From}), recipient's address (\texttt{To}), value (\texttt{Amount}), millisecond timestamp (\texttt{Timestamp}), transaction class (\texttt{Type} as \texttt{"transfer"}), and current account sequence number (\texttt{Nonce}), are structured into a map. In \texttt{SigningBytes()}, this payload is serialized into JSON. Go's standard JSON library deterministically marshals the map keys in alphabetical order, producing a canonical byte array:
  \begin{verbatim}
payload = {"amount":A, "from":F, "nonce":N, 
"timestamp":T, "to":R, "type":"transfer"}
  \end{verbatim}
  \item \textbf{Hashing the Payload:} The canonical JSON byte sequence is hashed using the SHA-256 algorithm from Go's standard \texttt{crypto/sha256} package, to produce a 32-byte message digest.
  \item \textbf{ECDSA Signature Generation:} The 32-byte digest is signed using the sender's private key via the standard \texttt{crypto/ecdsa} package. This produces two coordinate integers, $(r, s)$. In \texttt{SignBytes}, these coordinates are encoded as fixed-width 32-byte integers and appended together to form a raw 64-byte cryptographic signature:
  \begin{equation*}
      \text{Sig} = \text{pad}_{32}(r) \parallel \text{pad}_{32}(s)
  \end{equation*}
  This 64-byte signature is hex-encoded into a 128-character string and stored in the transaction's \texttt{Signature} field.
  \item \textbf{Public Key Attachment:} The sender's uncompressed ECDSA public key,represented in the standard uncompressed format (65 bytes: a \texttt{0x04} prefix byte followed by the 32-byte $X$ and $Y$ coordinates), is hex-encoded into a 130-character string and stored in the transaction's \texttt{PublicKey} field.
  \item \textbf{Transaction ID Computation:} Once fully signed, the unique transaction identifier (\texttt{ID}) is computed in\\ \texttt{CalculateTransactionID} by taking the SHA-256 hash of a pipe-delimited string of all fields, including the public key and signature:
  \begin{verbatim}
record = "From|To|Amount|Timestamp|Type|Nonce|
PublicKey|Signature"
tx.ID = Hex(SHA-256(record))
  \end{verbatim}
\end{enumerate}


\subsection{Transaction Verification Workflow}
When a node receives a transaction (either through the REST API or propagated via the P2P network), it runs stateful and stateless checks in \texttt{ValidateTransaction} before staging it in the mempool:
\begin{enumerate}
  \item \textbf{Address Derivation Check:} The node decodes the transaction's hexadecimal \texttt{PublicKey} and verifies that it derives the sender's address. In \texttt{AddressFromPublicKey}, the 65-byte uncompressed public key is hashed using SHA-256, and the first 20 bytes of the digest are hex-encoded to match the \texttt{From} address:
  \begin{equation*}
      \text{DerivedAddress} = \text{Hex}(\text{SHA-256}(\text{PublicKey})_{0\dots 19})
  \end{equation*}
  If the derived address does not match \texttt{tx.From}, the transaction is rejected.
  \item \textbf{Signature Validation:} The node parses the 64-byte signature back into $r$ and $s$ components, reconstructs the identical canonical signing JSON bytes, hashes them with SHA-256, and validates the signature against the decoded P-256 public key using \texttt{ecdsa.Verify}. Any corrupted signature (tested by XOR-ing bits) causes immediate rejection.
  \item \textbf{Stateful Sequence and Balance Verification:} The transaction's \texttt{Nonce} is compared against the expected next sequence number for the sender's address ($\text{NextNonce} = \text{LastNonce} + 1$). The transaction amount is also checked against the sender's accumulated ledger balance. If the nonce is incorrect or if the sender has insufficient funds (an overdraft attempt), the transaction is immediately discarded.
\end{enumerate}

\subsection{Chain Validation}\label{subsec:chain_validation}
To guarantee that all nodes arrive at an identical ledger state without a central coordinator, the \texttt{ValidateChain} function in \texttt{blockchain.go} rebuilds the entire ledger state in memory starting from the genesis block. This process tracks account balances (\texttt{balances map[string]int}), account nonces (\texttt{nonces map[string]uint64}), and unique transaction IDs (\texttt{seenTxIDs map[string]bool}) through the following sequential validation pipeline:

\begin{enumerate}
  \item \textbf{Genesis Validation:} The process verifies that the first block (index $0$) has a sequential index and a valid PoW hash prefix. It invokes \texttt{ValidateGenesisTransactions} to ensure all transactions in the genesis block are of type \texttt{"genesis"} with empty sender addresses and nonces of $0$. Its allocations are then applied to initialize account balances.
  
  \item \textbf{Header Sequencing and Hash Chaining:} For every subsequent block, the stateless \texttt{IsBlockValid} helper verifies that the block index is contiguous ($\text{curr.Index} = \text{prev.Index} + 1$), the \texttt{PrevHash} matches the preceding block's hash, and the current hash is correctly computed and meets the PoW difficulty target. Additionally, it ensures the timestamp is monotonically increasing and within a 2-hour future window limit.
  
  \item \textbf{Stateful Transaction Validation:} Transactions inside each block are validated against the accumulated ledger state using \texttt{ValidateBlockTransactionsAgainstState}. This step enforces all stateless and stateful validations, including ECDSA signature verification, address derivation, nonce sequencing, and balance/overdraft checks as detailed in the \textit{Transaction Verification Workflow} subsection. It also guarantees double-spend protection by rejecting duplicate transaction IDs across the chain. Additionally, coinbase transactions are verified to ensure exactly one reward transaction is present per block, positioned first, with an empty sender and nonce, and not exceeding the configured \texttt{MiningReward}.
  
  \item \textbf{State Application:} If all validations succeed, the transacted amounts are subtracted from the senders, added to the recipients, the account nonces are updated, the transaction IDs are marked as seen, and the loop proceeds to the next block until the entire chain is verified as valid.
\end{enumerate}


\subsection{Peer-to-Peer Protocol}
Decentralized communication is handled by a custom peer-to-peer networking implementation. The architecture employs a dual-protocol approach to separate node discovery from reliable data exchange, a three-stage progression visualized in Figure~\ref{fig:p2p_protocol}.
\begin{figure}[ht]
    \centering
    \includegraphics[width=1\linewidth]{Image/network.png}
    \caption{The three-stage peer-to-peer networking protocol, illustrating UDP peer discovery, TCP handshakes for transitive routing, and the inventory-based TCP gossip protocol.}
    \label{fig:p2p_protocol}
\end{figure}

Decentralized communication is handled by a custom peer-to-peer networking implementation. The architecture employs a dual-protocol approach to separate node discovery from reliable data exchange. Local peer discovery is facilitated by lightweight UDP broadcasts emitted every three seconds. When a node detects a broadcast from an unknown peer operating on the same network identifier, it initiates a reliable TCP connection.

Upon connection, nodes execute a strict \texttt{handshake} protocol. During this initial exchange, they verify network compatibility, such as matching Genesis hashes, and share state metadata, including their current chain height. To promote network robustness and transitive discovery, the handshake payload also includes up to three randomized, known peer addresses. This enables newly joined nodes to rapidly build a localized topology without relying on a central directory. To mitigate resource exhaustion and prevent network flooding, nodes strictly enforce a connection limit of six simultaneous peers.

Data propagation utilizes an efficient, inventory-based gossip protocol formatted as newline-delimited JSON. Rather than blindly broadcasting large payloads, nodes first announce the availability of new blocks or transactions via an \texttt{inv} (inventory) message containing only the respective cryptographic IDs. The receiving node cross-references these IDs against an internal, thread-safe \texttt{SeenTxs} and \texttt{SeenBlocks} cache. If the item is unknown, the node explicitly requests the full payload via a \texttt{getdata} message, subsequently receiving the actual \texttt{tx} or \texttt{block} object. Furthermore, for initial synchronization or resolving chain forks, a node discovering that its peer has a higher chain height during the handshake will issue a \texttt{getchain} request to download and evaluate the peer's full ledger.

\subsection{HTTP REST API}

To bridge the internal peer-to-peer network with external clients, each blockchain node exposes a comprehensive HTTP REST API. This interface serves as the primary gateway for user interactions, effectively decoupling the underlying Go backend from the user interface. The API provides read-only endpoints for querying the global state, such as fetching the current chain, checking wallet balances, and inspecting the mempool. Additionally, it exposes state mutating endpoints that allow clients to submit signed transactions (\texttt{POST /transactions}), request pre-mined tokens (\texttt{POST /faucet}), and trigger the Proof-of-Work mining process (\texttt{POST /mine}).

\subsection{Frontend Explorer and Security Lab}

Interfacing directly with the REST API is \textit{BlockView}, a Single-Page Application built with React and Vite. The frontend is designed to connect to one or more node APIs simultaneously, providing users with a real-time, interactive dashboard of the decentralized network. The application architecture is designed around network monitoring, chain visualization, and active node control.

Through the interface, users can monitor the live status of connected nodes via automatic polling, tracking metrics such as current chain height, mempool size, and active peer counts. The ledger itself is rendered as an interactive visual chain, allowing users to inspect detailed block metadata and individual transaction payloads. To facilitate active network participation, the application provides dedicated modules for requesting faucet tokens, constructing and signing ECDSA transfer transactions, and initiating mining operations.

Furthermore, the frontend incorporates an isolated \textit{Security Lab}. This specialized testing interface allows users to programmatically execute adversarial simulations including replay attacks, signature forgery, and state overdrafts against the connected node. The Security Lab visualizes the step-by-step execution flow of each attack, explicitly highlighting the cryptographic defense mechanism triggered and the actual error response returned by the backend system.


\section{Evaluation}

\subsection{Experimental Setup}

All experiments were conducted on a local machine running multiple blockchain node instances. Each node was started as a separate process with distinct API and P2P ports. The default configuration uses difficulty 3, a mining reward of 50 coins, and genesis allocations of 1000 (faucet), 100 (Alice), 50 (Bob), and 25 (Charlie) coins. The Go implementation was compiled with Go 1.22, and the frontend was served via Vite's development server.


\subsection{Experiment 1: Chain Integrity (Tampering Detection)}

\textbf{Objective.} The objective of this experiment is to verify the tamper-evident properties of the blockchain structure and evaluate the system's ability to detect unauthorized modifications to historical block data. Specifically, we test whether the system rejects a chain containing a block whose transaction payload was modified post-mining.

\textbf{Experimental Setup.} To simulate a data tampering attack, we initialized a local blockchain and mined a sequence of blocks containing valid transactions. Once the blocks were successfully appended to the ledger and their Proof-of-Work (PoW) consensus headers were established, we programmatically accessed an intermediate block's transaction array. We modified one transaction's transfer amount (altering the value from a valid transfer to a higher amount) to simulate an attacker trying to falsify historical ledger records. We then re-ran the node's full validation routine, \texttt{ValidateChain}, over the modified blocks to observe whether the tampering was detected and which checks triggered the failure.

\textbf{Results.} Upon executing the validation pipeline, the system immediately detected the alteration and flagged the entire chain as invalid. The validation execution was halted, and the \texttt{ValidateChain} function returned the error: \texttt{"block 1 ogiltigt: blockets hash är ogiltig"} (or \texttt{"invalid block hash"} in the user-facing test client). The system successfully prevented the modified block from being accepted or propagated further.

\textbf{System Protection Mechanisms.} The system's resilience to this attack is rooted in the layered validation pipeline described in the Chain Validation subsection (Section~\ref{subsec:chain_validation}). It relies on the following cascading defense mechanisms:
\begin{enumerate}
    \item \textbf{Deterministic Cryptographic Chaining:} As detailed in Section~\ref{subsec:chain_validation}, each block's header hash is a SHA-256 digest computed over its complete serialized representation, including its index, timestamp, list of transactions, previous hash, nonce, and difficulty. Because the transaction data is serialized directly into the block record, any modification to a transaction alters the input payload of the hash function. The stateless helper \texttt{IsBlockValid} recalculated the hash of the modified block and found that it no longer matched the hash stored in the block's header, triggering the initial failure.
    
    \item \textbf{Cascading Validation Failures:} Even if an attacker re-mined the tampered block to compute a new hash that matches its modified content (satisfying the Proof-of-Work difficulty target), the attack would still fail. The next block in the sequence stores the cryptographic hash of its predecessor in its \texttt{PrevHash} field. Since the tampered block's hash has changed, the next block's \texttt{PrevHash} would no longer match. The sequencing check $\text{curr.PrevHash} == \text{prev.Hash}$ would fail, causing a cascading validation breakdown that invalidates all subsequent blocks.
    
    \item \textbf{PoW Computational Barrier:} To make a tampered chain valid, an attacker would be forced to re-mine not only the tampered block but also every subsequent block in the chain to update all \texttt{PrevHash} links and satisfy the Proof-of-Work difficulty requirements. The computational work required grows exponentially, providing a physical security barrier against historical tampering.
    
    \item \textbf{State Reconstruction and Nonce Integrity:} Finally, even if an attacker possessed the computational hashing power to re-mine all subsequent blocks, any arbitrary modification to transaction data would conflict with the stateful validation steps (Section~\ref{subsec:chain_validation}). The \texttt{ValidateChain} function rebuilds the entire ledger state in memory from the genesis block, recalculating balances and verifying transaction signatures. Any unauthorized transaction values or wrong sequence numbers (nonces) would be rejected during stateful validation because the original sender's ECDSA signature would no longer match the modified payload.
\end{enumerate}

\subsection{Experiment 2: Transaction Validation (Attack Resistance)}

We implemented a comprehensive attack suite (\texttt{attack\_api.go}) that runs five distinct attack scenarios against an isolated blockchain instance. Each attack tests a specific security mechanism:

\begin{table}[ht]
\centering
\begin{tabular}{l l l}
\hline
Attack & Defense Mechanism & Result \\
\hline
Replay & Duplicate TX ID check & Blocked \\
Invalid Signature & ECDSA P-256 verification & Blocked \\
Wrong Public Key & Address derivation check & Blocked \\
Overdraft & State-based balance check & Blocked \\
Future Timestamp & 2-hour timestamp limit & Blocked \\
\hline
\end{tabular}
\caption{Attack resistance test results. All five attacks were correctly rejected by the system.}
\end{table}

\textbf{Replay Attack.} A previously broadcast, validly signed transaction was captured and submitted a second time. While the cryptographic signature remained valid, the transaction was rejected with \texttt{"transaction already exists in mempool"}. Because each transaction includes a unique, incrementing nonce, its deterministic SHA-256 hash (the Transaction ID) remains identical. The node's stateful \texttt{SeenTxs} tracking successfully identified the duplicate hash and dropped the request.

\textbf{Invalid Signature.} A valid transaction payload was intercepted, and its ECDSA signature was corrupted by XOR-ing all bytes with \texttt{0xFF} to simulate tampering. The system rejected the transaction with \texttt{"invalid signature"}. This confirms the integrity of the P-256 ECDSA implementation, any alteration to the signature or the underlying payload breaks the mathematical relationship between the message hash, the public key, and the $(r,s)$ signature pair.

\textbf{Wrong Public Key.} In an attempt to spoof an identity, a transaction was constructed claiming Eve's sender address but was signed using Alice's private key. The node rejected it because the validation layer derives the sender's address directly from the provided public key via a SHA-256 hash. Since Alice's private key cannot generate a valid signature for a payload tied to Eve's public key, the cryptographic verification failed (\texttt{"invalid signature"}), preventing unauthorized fund transfers.

\textbf{Overdraft.} A transaction attempting to send \texttt{balance + 9999} coins was constructed and signed perfectly. However, it was rejected with \texttt{"insufficient balance"}. This demonstrates that cryptographic validity alone is insufficient, the node continuously rebuilds the ledger state from the genesis block to track exact account balances. The state-based validation successfully prevented the creation of tokens out of thin air.

\textbf{Future Timestamp.} A block was constructed with a timestamp 3 hours in the future and successfully mined with a valid Proof-of-Work nonce. Despite the valid PoW, the receiving node rejected it with \texttt{"block timestamp is too far in the future"}. This validates the 2-hour timestamp window rule, a critical consensus mechanism designed to prevent malicious miners from manipulating network time to disrupt chain ordering, while still allowing for natural clock drift across the decentralized P2P network.

\subsection{Experiment 3: Multi-Node Synchronization}

We started three nodes on separate ports and verified synchronization behavior:
\begin{enumerate}
  \item Nodes discovered each other via UDP broadcast within 3 seconds.
  \item A transaction submitted to Node 1 appeared in Node 2 and Node 3's mempools via the inv/getdata protocol.
  \item A block mined on Node 1 was propagated to and accepted by all other nodes.
  \item A new node joining after block creation correctly synced the full chain via the getchain/chain protocol.
  \item Peer discovery was transitive: Node 3, which only directly discovered Node 2, learned about Node 1 through the handshake's \texttt{known\_peers} field.
\end{enumerate}

\section{Discussion}

\subsection{Implementation Choices and Educational Trade-offs}
Designing a decentralized blockchain for an educational and experimental context introduces a fundamental engineering challenge: balancing the technical realism of a production-like system with the need for operational clarity, resource containment, and observable state transitions. This tension directly addresses our first research question regarding the architectural trade-offs necessary to preserve instructional utility without compromising security guarantees.

Our design navigates these trade-offs through two primary implementation choices: the hybrid token distribution model and the REST-triggered mining model.

\subsubsection{Token Distribution: Pre-Mining vs. Dynamic Faucet}
In a public production blockchain like Bitcoin, initial liquidity and distribution are driven by continuous block rewards. However, in a sandboxed, multi-node lab environment, relying purely on mining to distribute initial tokens introduces a bootstrap deadlock: nodes cannot submit transfers because they possess $0$ coins, and they cannot acquire coins without mining, which requires an active mempool of transactions. 

To resolve this trade-off between conceptual correctness and immediate operational utility, we implemented a hybrid distribution system:
\begin{itemize}
    \item \textbf{Genesis Pre-mining:} We pre-mined a set of core testing accounts (\texttt{Alice}, \texttt{Bob}, \texttt{Charlie}) and a dedicated \textit{Faucet Wallet} directly in the genesis block, minting $1000$ coins. This provides immediate, out-of-the-box liquidity for simple peer-to-peer demonstration runs.
    \item \textbf{On-Demand Faucet:} To accommodate new, dynamically generated client wallets created in the React interface, we exposed a \texttt{POST /faucet} endpoint. This dynamically allocates $10$ tokens to newly booted addresses.
    \item \textbf{Educational Realism vs. Shortcut Simplification:} A major architectural temptation in educational projects is to shortcut transaction verification by simply incrementing account balances directly in the local state database. We rejected this shortcut because it violates state transition immutability and is technically unrealistic. Instead, the faucet behaves as a legitimate network participant. When triggered, the node constructs, signs, and broadcasts a standard transfer transaction from the pre-mined Faucet Wallet. This transaction enters the mempool and undergoes the exact same ECDSA signature verification, address derivation check, nonce sequencing, and P2P gossip propagation as any user transaction. This preserves absolute technical realism while maintaining a highly flexible and dynamic interface for students.
\end{itemize}

\subsubsection{Consensus Trigger: Manual vs. Automatic Mining}
Production blockchain networks employ continuous, automated mining, where miners race in background loops to find nonces. For an interactive laboratory prototype, this model fails to balance host preservation with pedagogical value. We opted instead for a \textbf{manual, HTTP-triggered mining model} (\texttt{POST /mine}), which represents a conscious trade-off:
\begin{itemize}
    \item \textbf{Host Preservation:} Running continuous SHA-256 mining on standard host machines (e.g., student laptops) would consume 100\% of available CPU resources, causing severe thermal throttling, loud fan noise, and rapid battery depletion.
    \item \textbf{Granular Observability of State Transitions:} In a continuous mining environment, the ledger state changes too rapidly for a human observer to inspect. By triggering mining manually via the REST API or the frontend React panel, students can stage multiple distinct transactions in the mempool, inspect their unconfirmed byte representations, and then initiate mining to witness the discrete state transition as transactions move from the mempool into a newly appended block.
    \item \textbf{Log Clarity and P2P Gossip Analysis:} Under manual mining, the peer-to-peer network is silent until a block is generated. This silence is crucial for debugging and analysis; when a block is mined, the stdout logs show a clear, linear progression of the gossip protocol—lightweight \texttt{inv} announcements, followed by selective \texttt{getdata} requests, full block transmission, and node-level validation. An automated loop would flood the console, obscuring the precise message exchanges and transitive routing behavior.
\end{itemize}

\subsection{Attack Resistance and Cryptographic Integrity}
Our experimental evaluation through the isolated Security Lab validates that even a simplified, single-process ledger implementation can achieve robust transaction-level security. The combination of NIST P-256 ECDSA signatures, address derivation, nonce ordering, and balance checks establishes a layered defense that effectively mitigates common blockchain vulnerabilities.

While these cryptographic safeguards provide robust transaction-level security—as empirically verified by the successful rejection of all simulated attack vectors in Experiment 2 (Table~1)—scaling this model to a live, decentralized environment exposes several systemic architectural limitations.

First, relying on sequential ECDSA signature verification inside \texttt{ValidateChain} introduces a severe computational throughput bottleneck due to repetitive elliptic curve point multiplications across all nodes. In production, resolving this strict CPU dependency requires transitioning to parallel validation architectures, hardware acceleration, or alternative cryptographic schemes like Schnorr or BLS that support signature aggregation.

Furthermore, the system suffers from network domain vulnerabilities; although nonce tracking secures the local ledger, the \texttt{SigningBytes()} payload in \texttt{transaction.go} lacks a cryptographic bind to the node's configured \texttt{NetworkID}. This absence of domain separation allows cross-chain replay attacks across parallel networks or forks, a vulnerability production frameworks mitigate by embedding unique chain identifiers directly into the signed data payload.

This scalability strain is further compounded by volumetric state bloat, as rebuilding and maintaining the active ledger state (\texttt{balances} and \texttt{nonces} maps) entirely within volatile RAM during \texttt{ValidateChain} execution is inherently unsustainable. As transaction and account volumes grow, production efficiency demands moving away from global in-memory maps toward persistent, disk-backed key-value datastores structured as authenticated Merkle Patricia Tries to optimize memory overhead.

Finally, the architecture exhibits an economic vulnerability to resource exhaustion because processing pending data within the mempool at zero marginal cost exposes the node to Denial of Service (DoS) attacks. Without an economic disincentive, such as transaction fees, to prevent malicious actors from cheaply flooding network buffers, production networks must address this structural vulnerability by implementing dynamic, fee-based prioritization markets to clear transaction queues.



\subsection{Networking Limitations and Gossip Overhead}
While our peer discovery functions well on a local network, relying on UDP broadcasts prevents discovery across network boundaries, as standard routers block broadcast traffic rather than the UDP protocol itself. Supporting wider networks would require alternative mechanisms, such as bootstrap node registries.
Furthermore, using ephemeral TCP connections for message propagation creates overhead compared to persistent connections. The system also lacks mechanisms for handling network partitions gracefully, which could lead to temporary chain forks that would need to be resolved via the longest-chain rule.

\subsection{Comparison to Production Blockchains}
Our prototype deliberately omits several key features found in enterprise blockchain networks, such as disk-backed key-value database persistence (e.g., LevelDB), Merkle trees for efficient transaction verification, dynamic difficulty adjustment, transaction fees, and smart contracts. 
However, by focusing purely on hash-linked blocks, Proof-of-Work mining, signed transactions, mempool staging, and gossip-based propagation, the system successfully reproduces the essential properties of a distributed ledger while maintaining an accessible codebase suitable for rapid educational experimentation.

\section{Related Work}

Bitcoin \cite{nakamoto2008bitcoin} introduced the first decentralized cryptocurrency, combining hash-linked blocks with Proof-of-Work consensus to achieve trustless agreement. Ethereum \cite{buterin2014ethereum} extended this model with a Turing-complete virtual machine enabling smart contracts. Our implementation draws inspiration from Bitcoin's UTXO-like balance computation and its gossip-based networking protocol (inv/getdata), while simplifying the consensus to basic longest-chain selection without difficulty adjustment.

Several educational blockchain implementations exist. Narayanan et al. \cite{narayanan2016bitcoin} provide a textbook treatment of Bitcoin's architecture but without a runnable system. Ivan Kuznetsov's ``Building Blockchain in Go'' tutorial \cite{kuznetsov2017blockchain} covers persistence and basic networking but omits ECDSA signing and attack testing. Our work differentiates itself by providing a complete, full-stack system with a modern frontend, automated security testing, and a P2P layer with UDP discovery, handshake-based peer exchange, and inventory-based propagation.

\section{Conclusion}

We have designed, implemented, and evaluated a modular blockchain system in Go with a React-based frontend explorer. The system faithfully implements the core mechanisms of decentralized ledger technology: SHA-256-based block linking, configurable Proof-of-Work consensus, ECDSA-signed transactions with nonce-based replay protection, a transaction mempool, and a peer-to-peer networking layer with UDP discovery and TCP communication.

Our evaluation demonstrates that: (1) mining time scales exponentially with PoW difficulty as predicted by theory; (2) hash-linked blocks provide effective tamper detection; (3) the combination of cryptographic signatures, address derivation, nonce ordering, balance validation, and timestamp checks successfully defends against five distinct attack vectors; and (4) multiple nodes can discover each other, synchronize chain state, and propagate transactions and blocks without centralized coordination.

The key insight from this work is that a relatively compact codebase (approximately 2,500 lines of Go plus 1,500 lines of React) can reproduce the essential security and consensus properties of production blockchain systems, while remaining transparent enough for educational analysis and experimentation.

\section{Future Work}

Several extensions would enhance the system's realism and capabilities:

\begin{itemize}
  \item \textbf{Persistent storage:} Adding disk-backed block storage using a key-value store (e.g., BoltDB) to enable node restarts without chain loss.
  \item \textbf{Dynamic difficulty adjustment:} Implementing automatic difficulty scaling based on recent block times, as in Bitcoin's 2,016-block adjustment window.
  \item \textbf{Merkle trees:} Adding Merkle root computation to block headers for efficient transaction inclusion proofs without downloading the full block.
  \item \textbf{Persistent P2P connections:} Replacing ephemeral TCP connections with long-lived connections to reduce handshake overhead and enable bidirectional communication.
  \item \textbf{Fork resolution:} Implementing explicit orphan block handling and fork detection to improve behavior under network partitions.
  \item \textbf{Transaction fees:} Adding fee-based transaction prioritization to the mempool to simulate economic incentives for miners.
\end{itemize}
